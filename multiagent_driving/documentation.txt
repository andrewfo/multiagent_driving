================================================================================
  multiagent_driving - Documentation
  Scripts: set_waypoints, track_navigator, websocket_server, websocket_client,
           car_filter, obstacle_detector & metrics_logger
================================================================================

This package contains ROS 2 scripts for autonomous lap driving on a physical
Roboracer car using the Nav2 navigation stack, with multi-agent coordination
over WebSocket.

  - set_waypoints:      Record waypoints by driving the car manually.
  - track_navigator:    Replay those waypoints autonomously using Nav2.
  - websocket_server:   Centralized relay server for multi-car state sharing.
  - websocket_client:   ROS 2 node bridging local car state to the server.
  - car_filter:         Filters lidar scans to prevent double-counting cars.
  - obstacle_detector:  Detects novel obstacles from lidar and shares them.

The typical workflow is: calibrate the car, build a SLAM map, record waypoints
with set_waypoints, then run track_navigator to drive laps autonomously.


================================================================================
  PREREQUISITES
================================================================================

Before using either script you need:

  1. A calibrated car (steering servo and speed gains configured).
  2. A SLAM map of your track saved to:
       ~/roboracer_ws/src/av_navigation/av_navigation/maps/<MAP_NAME>.yaml
     To create a map:
       cd ~/roboracer_ws/tmux/record_map/ && tmuxinator
     Drive slowly around the track, then save:
       export MAP_NAME=track
       ros2 run nav2_map_server map_saver_cli -f \
         ~/roboracer_ws/src/av_navigation/av_navigation/maps/${MAP_NAME}

  3. The Nav2 navigation stack running:
       cd ~/roboracer_ws/tmux/navstack/ && MAP=track tmuxinator
     OR:
       ros2 launch av_navigation navigation.launch.py map:=track

  4. nav2.yaml configured with your car's minimum turning radius:
       min_turning_r: <your_measured_radius>
       minimum_turning_radius: <your_measured_radius>
     And twist_to_ackermann.launch.py updated:
       max_curvature: <1 / your_turning_radius>

  5. Lidar publishing on /scan at >= 10 Hz.
  6. VESC driver publishing odometry.
  7. Static transforms active.

  8. Initial pose set in RViz2:
       - Connect via TurboVNC
       - Click "2D Pose Estimate" in RViz2
       - Click and drag on the map to place the car's starting position


================================================================================
  SCRIPT 1: set_waypoints
================================================================================

PURPOSE
  Records waypoints from the car's live AMCL localization pose while you drive
  it manually around the track. When you stop recording (Ctrl-C), it writes the
  waypoints directly into track_navigator.py, updating the TRACK_WAYPOINTS_XY
  list so track_navigator is ready to use immediately.

HOW IT WORKS
  - Subscribes to /amcl_pose (published by the AMCL localization node).
  - At a fixed interval (default: every 3 seconds), samples the car's current
    (x, y) position and appends it to an internal list.
  - On Ctrl-C, overwrites the TRACK_WAYPOINTS_XY variable in track_navigator.py
    with the recorded coordinates.

USAGE

  Basic (sample every 3 seconds):

    ros2 run multiagent_driving set_waypoints

  Custom interval (sample every 2 seconds):

    ros2 run multiagent_driving set_waypoints --interval 2

  Custom interval (sample every 5 seconds):

    ros2 run multiagent_driving set_waypoints --interval 5

PARAMETERS

  --interval <seconds>   Time between waypoint samples. Default: 3.0
                          Shorter intervals = more waypoints = smoother path.
                          Longer intervals = fewer waypoints = simpler path.

STEP-BY-STEP

  1. Make sure the navigation stack is running and the initial pose is set
     in RViz2 (see Prerequisites above).

  2. SSH into the car and start the script:
       cd ~/roboracer_ws && ./container shell
       ros2 run multiagent_driving set_waypoints --interval 3

  3. Drive the car manually around the track at a steady pace.
     The terminal will show each recorded waypoint:
       [INFO] [waypoint_recorder]: [1] Recorded waypoint (1.23, -0.45)
       [INFO] [waypoint_recorder]: [2] Recorded waypoint (2.10, -0.82)
       ...

  4. Once you have completed a full lap, press Ctrl-C.
     The script will save all waypoints into track_navigator.py and print
     the full waypoint list to the terminal.

  5. Rebuild the package so the updated track_navigator.py is installed:
       cd ~/roboracer_ws
       colcon build --packages-select multiagent_driving
       source install/setup.bash

TIPS
  - Drive at a slow, consistent speed for clean waypoints.
  - If you see "No pose received yet on /amcl_pose", the localization node
    is not running or the initial pose has not been set.
  - You can run the script multiple times; each run fully replaces the
    previous waypoints.
  - After saving, review the waypoints in track_navigator.py to remove any
    outliers or duplicates if needed.


================================================================================
  SCRIPT 2: track_navigator
================================================================================

PURPOSE
  Drives the car autonomously around the track in a loop using Nav2's
  FollowWaypoints action. It sends the waypoints defined in TRACK_WAYPOINTS_XY
  to the Nav2 waypoint follower, which plans and executes the path through
  each waypoint in order. The car will drive continuous laps until stopped.

HOW IT WORKS
  - Connects to the Nav2 FollowWaypoints action server.
  - Builds PoseStamped messages from the TRACK_WAYPOINTS_XY list.
  - Yaw at each waypoint is auto-computed to point toward the next waypoint.
  - Sends the full waypoint list as a goal; Nav2 plans the path.
  - After completing all waypoints (one lap), it sends them again for the
    next lap. Repeats until NUM_LAPS is reached or Ctrl-C is pressed.

USAGE

    ros2 run multiagent_driving track_navigator

  There are no command-line arguments. Configuration is done by editing
  the script directly (see Configuration below).

CONFIGURATION (edit track_navigator.py)

  TRACK_WAYPOINTS_XY   List of (x, y) tuples in the map frame.
                        Set these by running set_waypoints first, or by
                        manually reading coordinates from RViz2 / /amcl_pose.

  NUM_LAPS             Number of laps to drive.
                        - Set to a number (e.g., 5) for a fixed lap count.
                        - Set to None to loop forever (default).

  Speed                Controlled by Nav2 configuration, not this script.
                        Edit vx_max in:
                          src/av_navigation/av_navigation/config/nav2.yaml

STEP-BY-STEP

  1. Make sure the navigation stack is running:
       cd ~/roboracer_ws/tmux/navstack/ && MAP=track tmuxinator

  2. Set the initial pose in RViz2:
       - Open RViz2 via TurboVNC
       - Click "2D Pose Estimate"
       - Click and drag on the map at the car's actual position

  3. Make sure TRACK_WAYPOINTS_XY has been populated (by running set_waypoints
     or by manually entering coordinates).

  4. Hold R1 on the PS4 controller (or put joystick in autonomous mode).

  5. Run the navigator:
       ros2 run multiagent_driving track_navigator

  6. The terminal will show progress:
       [INFO] [track_navigator]: Waiting for Nav2 FollowWaypoints server...
       [INFO] [track_navigator]: Server ready. Starting autonomous laps.
       [INFO] [track_navigator]: --- Lap 1 ---
       [INFO] [track_navigator]:   -> waypoint 1/12
       [INFO] [track_navigator]:   -> waypoint 2/12
       ...
       [INFO] [track_navigator]: Lap 1 complete! All waypoints reached.
       [INFO] [track_navigator]: --- Lap 2 ---
       ...

  7. Press Ctrl-C at any time to stop.

TROUBLESHOOTING

  "Goal rejected by Nav2"
    - The navigation stack is not running, or the initial pose was not set.
    - Make sure you clicked "2D Pose Estimate" in RViz2 first.

  "Missed waypoints: [3, 7]"
    - Nav2 could not reach those waypoints (obstacle in the way, waypoint
      off the map, or the car couldn't plan a path).
    - Re-record waypoints or manually adjust the problematic entries.

  Car does not move after goal is accepted
    - Make sure R1 is held on the PS4 controller.
    - Check that the VESC driver is running and publishing odometry.

  Car oscillates or drives erratically
    - Waypoints may be too close together or positioned off the drivable area.
    - Increase the --interval when recording, or thin out waypoints manually.
    - Check that nav2.yaml turning radius matches your car's actual capability.


================================================================================
  SCRIPT 3: websocket_server
================================================================================

PURPOSE
  A centralized WebSocket relay server that receives JSON state messages from
  each connected car and broadcasts them to all other cars. Run this on one
  car or on a laptop that is on the same network as all participating cars.

HOW IT WORKS
  - Listens for incoming WebSocket connections on a configurable host/port.
  - When a car connects, it immediately receives the last-known state of
    every other car already in the session (no stale start).
  - Each incoming JSON message is cached (keyed by car_id) and broadcast to
    all other connected peers.
  - When a car disconnects its cached state is removed so remaining cars
    no longer see a phantom obstacle.

PROTOCOL
  Two message types flow through the server:

  State message (car -> server -> all peers):

    {
      "car_id":       "car1",           // set via car_id ROS param at launch
      "x":            1.234,            // map-frame position
      "y":            5.678,
      "yaw":          0.7854,           // heading in radians
      "vx":           0.35,             // linear velocity (m/s)
      "status":       "navigating",     // "navigating" or "idle"
      "current_goal": {"x": 3.0, "y": -2.0},  // current goal or null
      "obstacles": [                    // optional local obstacle detections
        {"x": 3.0, "y": 4.0},
        {"x": 3.5, "y": 4.2}
      ]
    }

  Command message (commander -> server -> target car only):

    {
      "type":       "command",
      "target_car": "car1",
      "action":     "navigate",         // action name
      "goal":       {"x": 3.0, "y": -2.0}  // action-specific payload
    }

  Command messages are routed only to the target car. The car's
  websocket_client publishes received commands on /commander_cmd (String).

USAGE

  Standalone (no ROS required):

    python3 websocket.py --host 0.0.0.0 --port 8765

  Via ros2 run (after colcon build):

    ros2 run multiagent_driving websocket_server
    ros2 run multiagent_driving websocket_server -- --port 9000

PARAMETERS

  --host <address>   Network interface to bind. Default: 0.0.0.0 (all).
  --port <number>    Port to listen on. Default: 8765.

STEP-BY-STEP

  1. Choose one machine to act as the server. It must be reachable by all
     cars on the network. Note its IP address (e.g., 192.168.1.100).

  2. Start the server:
       ros2 run multiagent_driving websocket_server

  3. You should see:
       WebSocket server listening on ws://0.0.0.0:8765

  4. On each car, start the websocket_client node (see Script 4).

  5. The server terminal will log connections and disconnections:
       [+] New connection (1 total)
       [+] New connection (2 total)
       [-] car2 disconnected (1 remaining)

  6. Press Ctrl-C to shut down the server.

TIPS
  - The server has no ROS dependencies and can run on any machine with
    Python 3 and the websockets library (pip install websockets).
  - Only one server should be running at a time on a given port.


================================================================================
  SCRIPT 4: websocket_client
================================================================================

PURPOSE
  A ROS 2 node that runs on each car and bridges local state to the
  centralized WebSocket server. It sends the car's own pose (and any local
  obstacle detections) to the server and publishes other cars' poses on ROS
  topics so the Nav2 costmap can treat them as obstacles.

HOW IT WORKS
  - Subscribes to amcl_pose to get this car's localized position.
  - Subscribes to /odom (Odometry) to read linear velocity (vx).
  - Optionally subscribes to /obstacles (PoseArray) for local obstacle data.
  - Sends a JSON state message to the server each time the car's pose updates.
    The message includes vx, status ("navigating"/"idle"), and current_goal.
  - Receives JSON messages from the server about other cars and publishes:
      /swarm_poses     (PoseArray) – other cars' positions and orientations.
      /swarm_obstacles (PoseArray) – obstacles reported by other cars.
  - Receives command messages (type: "command") from the server and publishes
    them on /commander_cmd (String) for a goal executor node to consume.
  - The SwarmLayer costmap plugin subscribes to /swarm_poses and marks those
    locations as lethal obstacles in the Nav2 costmap.
  - Automatically removes stale neighbors not heard from in over 2 seconds.
  - If the connection drops, the node automatically reconnects every 5 s.

USAGE

    ros2 run multiagent_driving websocket_client \
      --ros-args -p car_id:=car1 -p server_ip:=192.168.1.100 -p server_port:=8765

  NOTE: Do NOT use -r __ns:=/<name> to distinguish cars. Namespacing causes
  relative topic names (amcl_pose, /follow_waypoints) to resolve incorrectly.
  Use the car_id parameter instead.

ROS PARAMETERS

  car_id       (string)  Unique identifier for this car (e.g. "car1", "car2").
                          Must differ between cars — the server uses it to route
                          state, and each node filters out messages with its own
                          car_id to avoid acting on its own broadcasts.
                          Default: "" (falls back to ROS namespace or
                          "car_default" — always set this explicitly).

  server_ip    (string)  IP address of the WebSocket server.
                          Default: "localhost"

  server_port  (int)     Port of the WebSocket server.
                          Default: 8765

SUBSCRIPTIONS

  amcl_pose       (PoseWithCovarianceStamped)  This car's localization.
  /odom           (Odometry)                   Odometry for linear velocity.
  /obstacles      (PoseArray)                  Local obstacle detections
                                                (optional, only sent if
                                                 a node publishes on this
                                                 topic).

PUBLICATIONS

  /swarm_poses    (PoseArray)   Poses of all other connected cars, in the
                                 map frame. Published at 10 Hz. Consumed by
                                 the SwarmLayer costmap plugin.

  /swarm_obstacles (PoseArray)  Obstacles reported by other cars, in the
                                 map frame. Published at 10 Hz. Can be
                                 consumed by an additional costmap layer or
                                 a planner node if desired.

  /commander_cmd  (String)      JSON command messages received from the
                                 centralized commander via the server.
                                 Consumed by a goal executor node.

STEP-BY-STEP

  1. Make sure the websocket_server is running on the chosen host and
     note its IP address (e.g., 192.168.1.100).

  2. Make sure the navigation stack and AMCL localization are running on
     the car (see Prerequisites).

  3. Start the client:
       ros2 run multiagent_driving websocket_client \
         --ros-args -p server_ip:=192.168.1.100

  4. The terminal will show:
       [INFO] [websocket_client_node]: [car1] WebSocket client targeting
              ws://192.168.1.100:8765
       [INFO] [websocket_client_node]: [car1] Connected to
              ws://192.168.1.100:8765

  5. Repeat on every car, giving each a unique car_id:
       ros2 run multiagent_driving websocket_client \
         --ros-args -p car_id:=car2 -p server_ip:=192.168.1.100

  6. Verify that /swarm_poses is being published:
       ros2 topic echo /swarm_poses

TROUBLESHOOTING

  "WebSocket error: [Errno 111] Connection refused"
    - The server is not running or is not reachable at the given IP/port.
    - Check that the server is started and firewall rules allow the port.
    - The node will keep retrying every 5 seconds.

  /swarm_poses is published but always empty
    - No other cars are connected to the same server.
    - Or the other cars have not set their initial pose in RViz2 yet
      (AMCL is not publishing amcl_pose).

  Cars see each other but Nav2 does not avoid them
    - Make sure the SwarmLayer costmap plugin is enabled in nav2.yaml.
    - See the swarm_costmap_plugin package documentation.


================================================================================
  SCRIPT 5: car_filter
================================================================================

PURPOSE
  Prevents Nav2 from double-counting other cars as unknown obstacles. Without
  this node, each car appears twice in the costmap: once from the lidar's
  obstacle layer (as an unknown obstacle) and once from the SwarmLayer (as a
  known car). The double-counting inflates the perceived obstacle size and can
  cause Nav2 to plan unnecessarily wide detours or fail to find a path.

  The car_filter node removes laser scan points that fall within a configurable
  radius of any known car position, then republishes the cleaned scan on
  /scan_filtered. The Nav2 obstacle layers read /scan_filtered instead of
  /scan, so they only see real obstacles (walls, cones, pedestrians) while the
  SwarmLayer handles car avoidance.

HOW IT WORKS
  - Subscribes to /scan (raw lidar), /swarm_poses (known car positions from
    the websocket_client), and amcl_pose (this car's localization).
  - For each laser scan, projects every scan point into the map frame using
    the car's current AMCL pose.
  - Checks each projected point against all known car positions. If a point
    falls within car_radius of any car, it is replaced with an out-of-range
    value (range_max + 1.0). This causes the obstacle layer to ignore the
    point while still allowing raytracing to clear the space behind it.
  - Publishes the filtered scan on /scan_filtered at the same rate as /scan.

USAGE

    ros2 run multiagent_driving car_filter

  With custom radius (default is 0.25 m):

    ros2 run multiagent_driving car_filter \
      --ros-args -p car_radius:=0.30

ROS PARAMETERS

  car_radius   (float)  Radius around each known car within which scan
                         points are filtered out. Default: 0.25 m.
                         Increase if cars still appear as ghost obstacles.
                         Decrease if real obstacles near cars are being
                         incorrectly filtered.

SUBSCRIPTIONS

  /scan         (LaserScan)                  Raw lidar data.
  /swarm_poses  (PoseArray)                  Other cars' poses from
                                              websocket_client (map frame).
  amcl_pose     (PoseWithCovarianceStamped)  This car's localization.

PUBLICATIONS

  /scan_filtered (LaserScan)  Cleaned scan with car-occupied points removed.
                               Consumed by the obstacle_layer (global costmap)
                               and voxel_layer (local costmap) in nav2.yaml.

NAV2 CONFIGURATION

  The car_filter node requires that the obstacle layers in nav2.yaml read
  /scan_filtered instead of /scan. This change has already been made:

    global_costmap → obstacle_layer → scan → topic: /scan_filtered
    local_costmap  → voxel_layer    → scan → topic: /scan_filtered

  The collision_monitor still reads the raw /scan topic for safety — it
  should always see all obstacles including other cars.

STEP-BY-STEP

  1. Make sure the navigation stack is running and AMCL is publishing
     amcl_pose (initial pose set in RViz2).

  2. Make sure the websocket_client is running and /swarm_poses is being
     published (at least one other car connected to the server).

  3. Start the filter node:
       ros2 run multiagent_driving car_filter

  4. Verify that /scan_filtered is being published:
       ros2 topic echo /scan_filtered --once

  5. The Nav2 costmap will now use the filtered scan automatically.

TROUBLESHOOTING

  Cars still appear as ghost obstacles in the costmap
    - Increase car_radius: --ros-args -p car_radius:=0.35
    - Check that /swarm_poses is being published and contains the other
      cars' poses: ros2 topic echo /swarm_poses
    - Check that amcl_pose is being published (initial pose must be set).

  Real obstacles near cars are being filtered out
    - Decrease car_radius: --ros-args -p car_radius:=0.20
    - This is a trade-off; the default 0.25 m works well for standard
      Roboracer cars (0.30 m long x 0.18 m wide).

  /scan_filtered is not being published
    - Check that /scan is being published by the lidar driver.
    - Check the node is running: ros2 node list | grep car_filter

  Nav2 is not using the filtered scan
    - Verify nav2.yaml has topic: /scan_filtered in both the obstacle_layer
      and voxel_layer sections.
    - Restart the navigation stack after changing nav2.yaml.


================================================================================
  SCRIPT 6: obstacle_detector
================================================================================

PURPOSE
  Detects novel (non-wall) obstacles from the filtered lidar scan and publishes
  their positions on /obstacles. This is the node that feeds the existing
  obstacle-sharing pipeline: other cars receive these obstacles via the
  websocket_client and mark them in their costmaps via the SwarmLayer plugin.

  Without this node, the /obstacles topic has no publisher and obstacle sharing
  between cars does not happen. Each car's lidar detections stay local only.

HOW IT WORKS
  - Subscribes to /scan_filtered (lidar with car points already removed by
    car_filter) and amcl_pose (this car's localization).
  - Subscribes to /map (OccupancyGrid) to learn the static environment.
  - On a timer (default 10 Hz), processes the latest scan:
      1. Converts scan polar coordinates to (x, y) points in the map frame
         using the car's AMCL pose.
      2. Filters out points that correspond to known walls using the static
         map 
      3. Clusters the remaining points using single-linkage Euclidean
         clustering (default threshold: 0.3 m).
      4. Discards clusters with fewer than min_cluster_size points (default: 3)
         to suppress noise.
      5. Computes the centroid of each surviving cluster.
  - Publishes centroids as a PoseArray on /obstacles (frame: map).

DATA FLOW

  /scan_filtered (lidar, cars removed)
       |
       v
  obstacle_detector_node
       |  - projects to map frame
       |  - filters out static-map walls
       |  - clusters remaining points
       |  - publishes centroids
       v
  /obstacles (PoseArray)
       |
       v
  websocket_client_node  ---->  WebSocket server  ---->  other cars
       |                                                      |
       v                                                      v
  (local car ignores                                 /swarm_obstacles
   its own obstacles                                      |
   from /swarm_obstacles)                                 v
                                                    SwarmLayer plugin
                                                         |
                                                         v
                                                  costmap lethal cells
                                                         |
                                                         v
                                                  Nav2 plans around them

USAGE

    ros2 run multiagent_driving obstacle_detector

  With custom parameters:

    ros2 run multiagent_driving obstacle_detector \
      --ros-args -p cluster_distance:=0.4 -p min_cluster_size:=5

ROS PARAMETERS

  cluster_distance     (float)  Maximum Euclidean distance between two scan
                                 points for them to be in the same cluster.
                                 Default: 0.3 m.
                                 Increase if obstacles are being split into
                                 multiple detections. Decrease if separate
                                 obstacles are being merged into one.

  min_cluster_size     (int)    Minimum number of scan points required for a
                                 cluster to count as an obstacle.
                                 Default: 3.
                                 Increase to suppress noise at the cost of
                                 missing small obstacles. Decrease to detect
                                 smaller objects.

  publish_rate         (float)  Rate (Hz) at which obstacle detections are
                                 published. Default: 10.0.
                                 The node caches the latest scan and processes
                                 it on the timer, so the actual output rate is
                                 min(publish_rate, scan_rate).

  map_occupied_thresh  (int)    OccupancyGrid cell value at or above which a
                                 cell is considered occupied (wall).
                                 Default: 50.
                                 The standard ROS convention is 0 = free,
                                 100 = occupied, -1 = unknown. Points landing
                                 on cells >= this threshold are discarded.

SUBSCRIPTIONS

  /scan_filtered  (LaserScan)                  Filtered lidar data (from
                                                car_filter node). Points near
                                                known cars already removed.

  amcl_pose       (PoseWithCovarianceStamped)  This car's localization, used
                                                to project scan points into
                                                the map frame.

  /map            (OccupancyGrid)              Static map used to filter out
                                                known walls and barriers.
                                                Subscribed with TRANSIENT_LOCAL
                                                durability so the latched map
                                                is received on startup.

PUBLICATIONS

  /obstacles      (PoseArray)   Centroid positions of detected novel obstacle
                                 clusters, in the map frame. Consumed by the
                                 websocket_client_node and forwarded to other
                                 cars via the WebSocket server.

STEP-BY-STEP

  1. Make sure the navigation stack is running and the initial pose is set
     in RViz2 (AMCL must be publishing amcl_pose).

  2. Make sure car_filter is running (so /scan_filtered is available).

  3. Start the obstacle detector:
       ros2 run multiagent_driving obstacle_detector

  4. You should see:
       [INFO] [obstacle_detector_node]: Static map received (384x384, res=0.050m)
       [INFO] [obstacle_detector_node]: Obstacle detector active — ...

  5. Verify obstacles are being detected:
       ros2 topic echo /obstacles

  6. Place a novel object (e.g., a box) on the track in front of the car's
     lidar. You should see poses appear on /obstacles corresponding to the
     object's position.

  7. To visualize in RViz2, add a PoseArray display on the /obstacles topic.

TROUBLESHOOTING

  /obstacles is always empty
    - No novel obstacles are visible to the lidar. Only objects that are NOT
      in the static map are published. Walls and barriers are filtered out.
    - Place a new object on the track to test.
    - Check that /scan_filtered is being published:
        ros2 topic hz /scan_filtered
    - Check that amcl_pose is being published (initial pose must be set).

  Walls are being published as obstacles
    - The static map may not match the current environment. Rebuild the map
      with SLAM if the environment has changed significantly.
    - Try increasing map_occupied_thresh if the map has low-confidence cells.

  Too many false detections (noise)
    - Increase min_cluster_size (e.g., 5 or 8) to require more scan points
      before reporting an obstacle.
    - Decrease cluster_distance to avoid merging noise with real detections.

  Obstacles are split into multiple detections
    - Increase cluster_distance (e.g., 0.5) so nearby points are grouped
      into a single cluster.

  Other cars do not see shared obstacles
    - Make sure websocket_client is running and connected to the server.
    - Check that the SwarmLayer plugin is enabled in nav2.yaml.
    - Verify /swarm_obstacles is being published on the other car:
        ros2 topic echo /swarm_obstacles


================================================================================
  SCRIPT 7: metrics_logger
================================================================================

PURPOSE
  Records collision-avoidance experiment metrics to a CSV file for post-run
  analysis. Designed to be run alongside track_navigator during Phase C
  experiments, comparing the swarm-comms system against a baseline.

  Three classes of safety events are logged:
    - near_miss:       Minimum lidar range dropped below near_miss_threshold.
                       Indicates a close encounter with a wall or object.
    - collision_proxy: Minimum lidar range dropped below collision_threshold.
                       Strong indicator of a physical collision (car physically
                       fit within ~0.2 m of an object).
    - lap_start /      The car entered the lap-start trigger zone. Used to
      lap_complete:    measure per-lap travel time.

HOW IT WORKS
  - Subscribes to /scan (raw LaserScan, before car_filter) so it captures all
    proximity events including to other cars.
  - Subscribes to amcl_pose to track the car's position for CSV context and
    for lap-boundary detection.
  - On each scan, computes the minimum valid range. If it falls below either
    threshold, writes a row to the CSV immediately.
  - Detects lap completion by monitoring proximity to a configurable trigger
    point (default: waypoint 0 at (-0.3, -0.25)). Each entry into the trigger
    zone counts as one lap crossing; a debounce prevents double-counting.
  - Every 10 seconds, logs a summary line with lap count and event totals.

USAGE

  Via launch file (recommended — works in any mode):

    # Baseline run (single car, no comms)
    ros2 launch multiagent_driving multiagent.launch.py \
      mode:=single log_metrics:=true \
      metrics_output_file:=/tmp/baseline_run1.csv

    # Swarm run (multi car, comms enabled)
    ros2 launch multiagent_driving multiagent.launch.py \
      mode:=multi server_ip:=<SERVER_IP> log_metrics:=true \
      metrics_output_file:=/tmp/swarm_run1.csv

  Standalone:

    ros2 run multiagent_driving metrics_logger \
      --ros-args -p output_file:=/tmp/run1.csv

ROS PARAMETERS

  near_miss_threshold  (float)  Min scan range (m) to log a near_miss event.
                                 Default: 0.35 m.

  collision_threshold  (float)  Min scan range (m) to log a collision_proxy
                                 event. Default: 0.20 m.

  lap_start_x          (float)  x coordinate of the lap-start trigger point
                                 in the map frame. Default: -0.3 (waypoint 0).

  lap_start_y          (float)  y coordinate of the lap-start trigger point.
                                 Default: -0.25 (waypoint 0).

  lap_trigger_radius   (float)  The car must come within this distance of the
                                 trigger point to count a lap. Default: 0.5 m.

  output_file          (string) Full path for the CSV output. If empty, a
                                 timestamped file is auto-created in /tmp.

SUBSCRIPTIONS

  /scan      (LaserScan)                  Raw lidar. Used for proximity events.
  amcl_pose  (PoseWithCovarianceStamped)  Car position for lap detection
                                           and CSV context columns.

CSV FORMAT

  timestamp_sec, event_type, value, car_x, car_y

  timestamp_sec  ROS clock time of the event (float, seconds).
  event_type     One of: near_miss | collision_proxy | lap_start | lap_complete
  value          near_miss/collision_proxy: min scan range (m).
                 lap_start: lap number (int).
                 lap_complete: lap duration in seconds (float).
  car_x / car_y  AMCL position at event time.

  Example rows:
    1743200100.123, near_miss,       0.3201, 2.145, -1.832
    1743200112.445, collision_proxy, 0.1874, 2.150, -1.830
    1743200145.002, lap_start,       2.0000, -0.291, -0.248
    1743200145.003, lap_complete,    47.231, -0.291, -0.248

STEP-BY-STEP

  1. Decide on experiment condition: single (baseline) or multi (swarm).

  2. Launch with log_metrics:=true and a unique output_file per run:
       ros2 launch multiagent_driving multiagent.launch.py \
         mode:=single log_metrics:=true \
         metrics_output_file:=/tmp/baseline_run1.csv

  3. Set initial pose in RViz2 on all participating cars.

  4. Start autonomous driving and let the cars complete several laps.

  5. Ctrl-C to stop. The CSV is flushed to disk on every event so no data is
     lost if the run is interrupted.

  6. Copy the CSV off the car:
       scp roboracer@<CAR_IP>:/tmp/baseline_run1.csv ./data/

  7. Analyze with any CSV tool. Key comparisons:
       - Count of near_miss rows: baseline vs swarm
       - Count of collision_proxy rows: baseline vs swarm
       - Mean/std of lap_complete values: baseline vs swarm

TROUBLESHOOTING

  CSV file is empty after run
    - Check that /scan is being published: ros2 topic hz /scan
    - Check that amcl_pose is being published (initial pose must be set).
    - Check the node is running: ros2 node list | grep metrics_logger

  Lap count is wrong or laps are double-counted
    - The trigger zone radius (lap_trigger_radius) may be too large, causing
      multiple triggers per crossing. Decrease it (e.g., 0.3 m).
    - The trigger point (lap_start_x/y) should match the car's actual start
      position on the track.

  near_miss events fire constantly (high noise)
    - Increase near_miss_threshold slightly (e.g., 0.40 m) if the track walls
      are closer than expected.
    - Check that /scan is the raw scan, not /scan_filtered.

  ros2 run multiagent_driving websocket_server

  ros2 run multiagent_driving websocket_client \
    --ros-args -p car_id:=car1 -p server_ip:=<SERVER_IP>
  # On the second car:
  # ros2 run multiagent_driving websocket_client \
  #   --ros-args -p car_id:=car2 -p server_ip:=<SERVER_IP>

  # 5. Start the car filter on every car (prevents double-counting)
  ros2 run multiagent_driving car_filter

  # 6. Start the obstacle detector on every car (shares novel obstacles)
  ros2 run multiagent_driving obstacle_detector

  # 7. Start autonomous driving on every car (hold R1 on controller)
  ros2 run multiagent_driving track_navigator

  # 8. (optional) Log experiment metrics
  ros2 run multiagent_driving metrics_logger \
    --ros-args -p output_file:=/tmp/swarm_run1.csv

  # 9. Ctrl-C to stop. Collect CSVs with: scp roboracer@<IP>:/tmp/*.csv ./data/
